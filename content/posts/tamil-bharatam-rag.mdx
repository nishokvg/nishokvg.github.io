## Problem 1: The Font Encoding Trap

Every RAG tutorial assumes text extraction is free. Paste a PDF path in, get clean text out. That assumption breaks the moment you touch a non-English document from before 2005.

These PDFs use **Bamini** — a 1990s Tamil font encoding invented before Unicode was adopted for Indian languages. Bamini maps its glyphs to the ASCII range. The letter **க** is stored as the byte `0x47` — which is `G` in ASCII. The entire font is a lookup table that only Bamini-aware renderers understand.

`pypdf` extracts what it sees: ASCII-range bytes. So does `pymupdf`. Neither knows about Bamini. The result:

```
# What the PDF contains (Bamini-encoded):
¬è Å¨Ä¿¨ÁÂ¢Ä Å¢üÅ¢¨Ä¡ö

# What it should say:
அர்ஜுனன் வில்லேந்தி நின்றான்
```

I spent three days on Bamini-to-Unicode mapping tables. They exist. They're incomplete. One character — the vowel carrier ர — doubles as a vowel modifier glyph in Bamini. Context-free mapping can't resolve it. Conversion accuracy was around 70%. In an 80-word chunk, that's 24 wrong characters. Embeddings built on that garbage are meaningless.

Two approaches actually work:

| Approach | How | Quality | Speed |
|----------|-----|---------|-------|
| Tesseract OCR | Renders page as image, ignores font | High | ~38 pages/min |
| Color metadata | Uses PyMuPDF span RGB values | High | 2177 pages in 30 seconds |

The color approach is the interesting one.

## The Insight Hidden in the Ink

I was staring at a photo of a Villi Bharatam page when I noticed something.

The poem verses are printed in **dark red or teal ink**. The prose commentary is printed in **black**. Section headers are in **blue**. The book was typeset with semantic color — every content type has its own ink.

PyMuPDF's `get_text("dict")` returns a `color` field for every text span as a 24-bit integer. I had been discarding it. It turns out that field is the key to the entire corpus.

```python
def classify_color(color_int):
    r = (color_int >> 16) & 0xFF
    g = (color_int >> 8)  & 0xFF
    b = (color_int)       & 0xFF

    if (r + g + b) < 60:
        return "commentary"   # black ink → prose explanation

    if b > r + 40 and b > g + 20:
        return "header"       # blue ink → section title

    return "verse"            # any real color → Villi's poem
```

Three lines of RGB arithmetic. No font decoding. No OCR. The text inside each span is still garbled Bamini bytes — but we now know *which* garbled bytes are poem lines and which are commentary. And that structure is exactly what we need to chunk and index intelligently.

Running this across all 2177 pages takes 28 seconds.

## The OCR Speed Wall

For chunks where I needed actual Unicode text (not just structure), I needed OCR. Tesseract with `lang="tam+eng"` reads the page as an image — it doesn't care about Bamini at all. It sees pixels, not bytes, and produces clean Unicode Tamil.

The first version was brutal:

```python
# Initial approach — load entire PDF at once
images = convert_from_path(pdf_path, dpi=200)  # 577 pages × 200dpi
# RAM: ~12GB just for ஆதி பருவம்
# Speed: 0.6 pages/min
```

0.6 pages per minute. At that rate, all 2177 pages would take **60 hours**.

The problem was memory: loading 577 high-DPI page images at once caused constant disk swapping. The fix was batch processing — load 10 pages, OCR them, discard the images, move on:

```python
# Fixed approach — batch of 10 pages
for start in range(0, total_pages, 10):
    images = convert_from_path(pdf_path, dpi=200,
                               first_page=start + 1,
                               last_page=min(start + 10, total_pages))
    for img in images:
        text = pytesseract.image_to_string(img, lang="tam+eng", config="--psm 1")
        # write to file, discard img immediately
    del images
```

Speed went from 0.6 pages/min to **38 pages/min** — a 63x improvement. All 2177 pages: ~57 minutes.

## Why BM25 Alongside FAISS?

Semantic search has a blind spot with proper nouns. When you ask *"கர்ணன் யார்?"*, the embedding of that question might cluster closest to chunks about general heroism or warriors — thematically related, but not mentioning Karna by name.

Tamil character names are exact tokens. *கர்ணன்*, *அர்ஜுனன்*, *துரியோதனன்* — these don't paraphrase. If the chunk doesn't contain the exact characters, BM25 won't return it. If the chunk does contain them, BM25 will rank it first regardless of semantic similarity.

**Reciprocal Rank Fusion** merges both lists without normalising scores:

```python
def rrf_merge(semantic_hits, keyword_hits, k=60):
    scores = {}
    for rank, idx in enumerate(semantic_hits):
        scores[idx] = scores.get(idx, 0) + 1 / (rank + k)
    for rank, idx in enumerate(keyword_hits):
        scores[idx] = scores.get(idx, 0) + 1 / (rank + k)
    return sorted(scores, key=scores.get, reverse=True)
```

A chunk that appears in both ranked lists gets a score bonus from both. In `--verbose` mode, chunks found by both methods are marked — those are the highest-confidence retrievals.

## The Embedding Timeout Marathon

OpenRouter's embedding API timed out. The first run: `batch_size=100`, `timeout=60s`. Died at batch 21 of 57.

Tamil Unicode is token-heavy. One Tamil word like *குருக்ஷேத்திர* becomes 12–18 tokens where the English "Kurukshetra" is 4. A batch of 100 Tamil chunks can hit 3x the token count of 100 English chunks. The API wasn't slow — it just had more work than the timeout allowed.

Fix path:

| Attempt | batch_size | timeout | Result |
|---------|-----------|---------|--------|
| 1 | 100 | 60s | Died at batch 21 |
| 2 | 20 | 120s | Died at batch 21 |
| 3 | 5 | 180s + retry | ✓ 1127 chunks in 9.4 min |

```python
for attempt in range(3):
    try:
        resp = requests.post(..., timeout=180)
        break
    except ReadTimeout:
        if attempt == 2:
            raise
        time.sleep((attempt + 1) * 5)
        print(f"  Batch {done} timeout, retrying in {wait}s...")
```

Two transient timeouts during the final run. Both recovered on the first retry.

## The Accuracy Results

I tested with 10 questions covering character identity, plot events, and relationships. Sample index: 30 pages per parva (~240 pages, ~14% of the full corpus).

| Backend | Model | Accuracy | Failure Mode |
|---------|-------|----------|-------------|
| Ollama local | llama3.1:8b | ~20% | Hallucinated answers confidently; ignored system prompt |
| OpenRouter | GPT-4o-mini | ~40–50% | Correctly refused when answer wasn't in retrieved chunks |

The GPT-4o-mini answers that were *wrong* were honest wrong — it said "I don't find this in the provided passages" instead of inventing. That's the right failure mode. The llama3.1:8b answers that were wrong were confident and fabricated. One answer described Karna's birth in detail — plausible-sounding Tamil mythology — from pages we had never indexed.

The real bottleneck isn't model quality. It's retrieval coverage. We indexed 240 of 2177 pages. A question about something that happens on page 400 of ஆதி பருவம் will never be answered correctly — we simply don't have that page in the index. GPT-4o-mini correctly identifies this gap. llama3.1:8b fills it with confabulation.

## Lessons I'd Carry Into the Next One

**Font encoding is a first-class problem in non-English RAG.** English RAG tutorials assume text extraction is free. For South Indian digitised texts from before 2005, budget a full day just on extraction before touching a single embedding. Bamini, ISCII, TAB encoding — they all require custom handling. There is no universal Tamil PDF extractor.

**Read the physical artefact, not just the file.** The Bamini problem was unsolvable at the byte level. The solution came from looking at a photograph of the actual book and noticing the ink colors. The metadata was there all along — I just wasn't looking in the right place.

**Batch size isn't about throughput — it's about token density.** The standard advice for embedding APIs is "use the largest batch size that fits in memory." That's English-centric advice. For token-heavy scripts (Tamil, Arabic, Chinese), start at 10% of the recommended batch size and scale up based on actual timeout behavior.

**A model that refuses honestly is better than a model that invents confidently.** llama3.1:8b at 20% accuracy feels like a working system — it always produces an answer. GPT-4o-mini at 40–50% feels more honest — it tells you when it doesn't know. For a literary reference tool, that honesty matters more than apparent completeness.

**Index coverage is the leverage point, not model choice.** Going from GPT-4o-mini to GPT-4 on a 14% corpus would improve answers on those 14%. Indexing the full 2177 pages with GPT-4o-mini would improve answers everywhere. Fix coverage first.

## What's Next

The path from 40% to 80%+ accuracy is mostly mechanical:

1. **Run OCR on all 2177 pages.** The pipeline exists. At 38 pages/min, that's ~57 minutes unattended.
2. **Rebuild the index on full corpus.** ~15,000 chunks instead of 1,127.
3. **Add parva-level routing.** Questions about the dice game route to சபா பருவம் (288 pages) instead of searching all 2177. Smaller search space, higher precision.
4. **Reranking.** Retrieve top-10 with FAISS+BM25, then pass all 10 to a cross-encoder to rerank to top-3. The cross-encoder reads the actual query against each chunk — it's much more accurate than embedding distance alone.
